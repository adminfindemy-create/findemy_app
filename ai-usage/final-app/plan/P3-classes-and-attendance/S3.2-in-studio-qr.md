# S3.2 ‚Äî In-studio QR check-in `[F13]` ‚Äî BUILD (all three sides)

> Feature Master Plan (master-execution-plan ¬ß3.2). Deps: **S3.1** (class detail entry point), **S1.3** (`mode`). Net-new on backend + both apps. Establishes the **shared `BatchAttendance` write foundation** that S3.3 (online auto-join) reuses.

## 1. Feature & intent
For an **in-studio** active class, the academy displays a **per-session QR**; the student **scans it to mark present**. **QR-only ‚Äî no manual marking** (locked, CLAUDE.md ¬ß7). Idempotent double-scan.

- **Refs:** master-feature-document ¬ß2.2 (QR attendance) / ¬ß4.5 (academy attendance-qr); prototypes academy `attendance-qr` (live "9 of 12 checked in") + student `checkin-scan` (camera ‚Üí "marked present").
- **Post-P2 reality (verified):** the `attendance` module is **manual-mark only** (`routes.ts` GET roster + POST `{date, marks}`; `service.ts:21`) ‚Äî **the manual POST + its academy toggle UI are removed**. `BatchAttendance` (`schema.prisma:684`) is `@@unique([batchId, userId, date])` (idempotent per day). **No QR, no scanner, no check-in token** exists. `lib/tokens.ts` has an ES256 JWT signer (reusable). **No camera dep** in student-app; **no QR-display dep** in academy-app.

## 3. The shared contract (lock first)
```
POST /studio/batches/:id/session/checkin-token   (academy auth)
  ‚Üí { token, expires_at }   // token = signed { batch_id, session_date(IST), kind:'checkin' }, TTL ~ session length
POST /attendance/checkin { token }               (student auth)
  ‚Üí verify token (sig + not expired) ‚Üí require enrolment + batch.mode == in-studio
  ‚Üí upsert BatchAttendance { batchId, userId, date: session_date, present: true, markedByAccountId: null }
  ‚Üí idempotent (double-scan = no-op via @@unique)
GET /studio/batches/:id/session/attendance?date=  (academy auth)
  ‚Üí { capacity, present_count, roster: [{ user_id, name, checked_in: bool, marked_at? }] }
```
- **Session = `(batch, date)`** (Decision 1 ‚Äî slots are gone). The token encodes the IST session date; the write keys off `(batchId, userId, date)` (existing unique ‚Üí idempotent). MVP assumes ‚â§1 in-studio session/day per batch.
- **Token is short-lived + session-scoped** (sign with the existing ES256 key, `lib/tokens.ts`; add `signCheckinToken`/`verifyCheckinToken`, or an HMAC over `${batchId}:${date}` + a server secret). It only proves "this is today's session for batch X" ‚Äî the **student's identity** comes from their auth on `/attendance/checkin` (so a leaked QR still only marks the scanning student, who must be enrolled).
- **QR-only:** the manual-mark endpoint + academy toggle UI are deleted; attendance for in-studio classes is exclusively student-scanned.

## 6. Sequencing & acceptance
1. Backend: `markedByAccountId` nullable (db push) ‚Üí token + checkin + roster; remove manual mark ‚Üí handoff. 2. Academy QR + live roster. 3. Student camera scan. 4. E2E.

**Definition of Done (E2E)**
- [ ] Academy opens an in-studio Guitar session ‚Üí **QR shown**; student taps "Scan to check in" ‚Üí scans ‚Üí **"marked present"**; academy roster flips them to **checked-in live**.
- [ ] **Double-scan is a no-op** (one present row).
- [ ] A non-enrolled or online-batch scan is **rejected**.
- [ ] **No manual-marking** endpoint/UI remains. Typecheck + full suite green; `db push` clean.
