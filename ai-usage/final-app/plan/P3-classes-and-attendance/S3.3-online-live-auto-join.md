# S3.3 — Online live class + auto-on-join `[F14]` — CHANGE / BUILD

> Feature Master Plan (master-execution-plan §3.2). Deps: **S3.1** (class detail), **S1.3** (`mode`), and the **S3.2 attendance foundation** (`BatchAttendance` nullable marker, `(batch,date)` key). Builds directly on the S2.1 `batch.id` room.

## 1. Feature & intent
For an **online** active class, the academy hosts a **real 100ms live room** ("Start live class"); the student joins (real A/V) and **joining marks attendance automatically** (no QR, no OTP) — **on the verified `peer.join` event**, not on screen-open. The academy sees a joined/not-joined roster.

> **DECISION (owner): build the real 100ms integration now — DF6 is un-deferred and folded into this slice.** This makes the auto-attendance *genuinely* join-gated (a 100ms webhook writes present), fixing the earlier "auto-on-open" overstatement. **Constraint:** like Razorpay, the integration is code-complete + typecheck-clean here, but **live A/V is verified once real 100ms credentials are set** (no keys → the existing `'mock-token'` fallback).

- **Refs:** master-feature-document §2.2 / §2.3 (online auto-on-join) / §4.6 (academy live-class); prototypes academy `live-class` host (Live pill, video stage, End class) + student join; CLAUDE.md §7 (online attendance = auto-on-join; Findemy-hosted room per batch).
- **Post-P2 reality (verified):** rooms are **keyed off `batch.id`** (`rooms/service.ts:45`, S2.1/D4); **"live" is timing-derived** (`rooms/repo.ts findLiveBatch`). `GET /rooms/batch/:id/token` (`routes.ts:5`) **writes NO attendance** today. **⚠️ `generateHmsToken` passes `batch.id` as the 100ms `room_id`** — but 100ms room ids are *their* generated ids, so no real room exists → it always falls back to `'mock-token'`. **No 100ms room provisioning, no webhook, no live start/end/roster.** `HMS_*` are read from `process.env` raw (not in the env schema). **SDK not installed** in either app; `student-app/app/live/[batch_id].tsx` is a stub; academy has no live screen. Razorpay's signed `POST /payments/webhook` is the pattern to mirror for the 100ms webhook.

## 3. The shared contract (lock first)
```
Batch.liveRoomId: String?   // the 100ms room id provisioned for this batch (lazy-created)

POST /studio/batches/:id/live/start (academy)  → ensure 100ms room exists → { host_token, room_id: liveRoomId }
GET  /rooms/batch/:id/token (student, REUSE)   → ensure room + gate on findLiveBatch → { token, room_id: liveRoomId, room_name }
POST /rooms/100ms/webhook (100ms → us, signed) → on 'peer.join.success': upsert BatchAttendance present
                                                 { batchId(from room metadata/userId), date: sessionDateIST(), present:true, markedByAccountId:null }
GET  /studio/batches/:id/live/roster (academy) → { joined: [{user_id,name,joined_at}], not_joined: [{user_id,name}] }
POST /studio/batches/:id/live/end (academy)    → 100ms disable/end room (optional) + record end
```
- **✅ Attendance = the REAL join (resolved overstatement).** A **signed 100ms webhook** fires `peer.join.success` when a peer actually connects to the room → the backend writes `BatchAttendance` present (server-authoritative — opening-and-leaving without connecting does **not** mark present). Map the joining peer → `(batchId, userId)` via the token's `user_id` + the room's batch (from `liveRoomId`). `date` = the shared **IST start-of-day** helper (same row as QR). Idempotent via `@@unique([batchId,userId,date])`. *(Fallback if webhooks aren't configured: a client `POST /rooms/batch/:id/joined` on the SDK's `onJoin` callback — client-trusted, less robust; prefer the webhook.)*
- **100ms room per batch.** A batch's room is provisioned **lazily** (first `live/start` or first student token) via the 100ms Management API (`POST /v2/rooms` with `HMS_TEMPLATE_ID`) and the id stored on `Batch.liveRoomId`; tokens are then minted for that real room. The template defines roles **`host`** (academy) + **`guest`** (student).
- **"Live" stays timing-derived** (`findLiveBatch`, S2.1/D4) — student joining is gated to the session window; "Start live class" is the academy entering the host room. `joined_at` comes from the webhook event time (or `markedAt`).
- **Roster:** active enrolments × today's present `BatchAttendance`. **One live session per batch at a time** (D4) — concurrent cohorts would need `batch.id + sessionStart`.
