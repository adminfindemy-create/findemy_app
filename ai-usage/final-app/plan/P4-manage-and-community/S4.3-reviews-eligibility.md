# S4.3 ‚Äî Reviews + eligibility (trial OR enrolment) `[F5]` ‚Äî CHANGE / BUILD

> Master ¬ßP4. Deps: **S2.3** (enrolments exist). Today reviews are structurally trial-only; this slice opens them to enrolled students and builds the student-side review entry (which doesn't exist at all).

## 1. Feature & intent
A student who has **taken a trial OR enrolled** at an academy can leave **one review** (rating + optional note); the academy responds. The academy's rating/count is canonical.

- **Refs:** master ¬ß3.10/¬ß4.8; gap-analysis (`Review.trialId` required+unique blocks enrolment reviews; no enrolled entry; 84-vs-32 count).
- **Post-P3 reality (verified):** `Review` (`schema.prisma:397-415`) has **`trialId String @unique` (required)** + `batchId`, **no `enrollmentId`/`source`**, **no `@@unique([userId,academyId])`**. `POST /reviews` gates on a trial only (`reviews/service.ts:9-13`) and writes the stale `Academy.ratingCount` cache (`:41`). **The student app has NO review-creation UI** ‚Äî neither post-trial nor enrolled (`program/[id]/review.tsx` is the enrol *payment* screen, not a review). Academy reviews list + respond are **aligned + REUSE** (`reviews.tsx`, `reviews/[id]/respond.tsx`). The **canonical count already exists**: `getReviewsSummary` groupBy (`studio/repo.ts:437-462`) computes count fresh ‚Äî the **84-vs-32 bug is the stale `Academy.ratingCount`** field.

## 3. The shared contract (lock first)
```
Review:  trialId String?   enrollmentId String?   source 'trial'|'enrollment'
         @@unique([userId, academyId])           // one review per (user, academy), either path

POST /reviews { academy_id, source:'trial'|'enrollment', batch_id, rating:1..5, comment?,
                trial_id? , enrollment_id? }
  ‚Üí eligibility: source='trial'  ‚Üí a Trial(userId, academyId, status = 'attended') exists
                 source='enrollment' ‚Üí an Enrollment(userId) whose batch.id = batch_id AND batch.academyId = academy_id exists
  ‚Üí reject if the (user, academy) review already exists (unique) ‚Üí 409 "already reviewed"
  ‚Üí recompute academy rating + count from ALL reviews (the groupBy) ‚Äî single source
```
- **One review per (user, academy).** Drop `trialId @unique`; add `@@unique([userId, academyId])`. `trialId`/`enrollmentId` both optional; exactly one set per `source` (validate in code ‚Äî Prisma can't express a conditional CHECK simply). **Product rule to lock:** a student reviews each academy **at most once**, whether they trialed *or* enrolled (a trial-then-enrol student keeps their one review). The student UI disables the entry once reviewed.
- **‚öÝÔ∏è `batchId` must be explicit (verified ambiguity).** `Review.batchId` is required, but a student can be enrolled in **multiple batches at one academy** (`@@unique([userId,batchId])` allows it) ‚Äî so `Enrollment.findFirst` can't pick the intended batch. **Require `batch_id` in the request** (the student is reviewing from a specific class/trial context) and verify it belongs to the academy + to the user's trial/enrolment. Never `findFirst`-guess the batch.
- **Trial eligibility = an *attended* trial** (`status = 'attended'`) ‚Äî a review reflects real experience. Not `booked` (hasn't happened), `missed` (no-show, never experienced the class), or `cancelled`. *(Owner decision 2026-06-28.)*
- **Count = the groupBy, on BOTH sides (verified gap).** `getReviewsSummary` (`studio/repo.ts:437-462`) is already canonical for the academy. **But the student-facing reads expose the stale `Academy.ratingCount`** ‚Äî `academies/service.ts:61` (search) + `:117` (detail) + the **sort at `:83`** (`rating √ó log(ratingCount+1)`) + `me/service.ts:31` (saved). Fixing only the academy summary leaves 84-vs-32 visible to students. Either (a) make these reads derive count from the review set, or (b) **keep `Academy.ratingCount` authoritative by recomputing it correctly from ALL reviews on every create** (the existing recompute is fine ‚Äî the risk is it drifting via deletes; if nothing deletes reviews, the cache stays correct). **Recommend (b)** ‚Äî cheapest, keeps the existing readers ‚Äî and add a note that any future review-delete path must recompute.

## 4. Component design, per side
- **Backend:** `schema.prisma` Review ‚Äî `trialId String?`, `enrollmentId String?`, **`source ReviewSource @default(trial)`** (the default backfills existing trial-only rows on `db push` ‚Äî required, else the NOT NULL add fails), drop `@unique` on trialId, add `@@unique([userId, academyId])`. **Pre-check the seed has no `(userId, academyId)` duplicate** before adding the unique constraint (verified: `seed.ts` loops one review per reviewer-per-academy ‚Üí safe). `reviews/service.ts createReview` ‚Äî take `batch_id`, branch on `source`, run the matching eligibility query (trial status = `attended`; enrolment by `batch_id`+academy), set `batchId` from the validated request, catch the unique violation ‚Üí `ALREADY_REVIEWED` 409, recompute `Academy.rating`+`ratingCount` from the **full** review set (no filtered query). Keep respond/read as-is; student count readers stay correct because the cache is recomputed on every create.
- **Student:** **BUILD** a review entry ‚Äî rating stars + optional chips/note, **no decision step** ‚Äî reachable from **class detail** (active enrolment ‚Üí `source:'enrollment'`) and the **post-trial** flow (`source:'trial'`). A `useCreateReview` hook ‚Üí `POST /reviews`. Show "You've already reviewed" when the unique gate trips.
- **Academy:** REUSE list + respond; the summary count now never disagrees with the list.

## 5. Sequencing & DoD
1. Backend: schema + eligibility + canonical count ‚Üí round-trip (trial review, enrolment review, duplicate rejected, non-eligible rejected). 2. Student: review entry (both sources). 3. E2E.

**Definition of Done (E2E)**
- [ ] An **enrolled, never-trialed** student posts a review; a **never-trial/never-enrolled** user is rejected; a **second** review by the same user for the same academy is rejected (409).
- [ ] Academy responds; the academy's **count is canonical** (no 84-vs-32 ‚Äî list length == summary count == displayed count).
- [ ] Existing trial reviews still work. Typecheck + suite green; `db push` clean.

## 6. Handoff
The `(user, academy)` uniqueness + `source` field. Flag `Academy.ratingCount` as a stale cache for the dead-code-registry if we stop writing it.
