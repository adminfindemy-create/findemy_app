# S5.4 — Dead-code removal + Prisma migration baseline — CHANGE (delete)

> Master §P5 (cleanup). This is **where the dead code gets removed** — the dedicated, pre-launch pass that executes [`../dead-code-registry.md`](../dead-code-registry.md) and resolves **DF5** (real migrations). Runs **last**, after every feature is built + verified.

## 1. Feature & intent
Generate a real **Prisma migration baseline**, then **delete the dead/dormant code** accumulated across phases — primarily the retired **Slot system** — in one coordinated pass that keeps all 5 packages green.

- **Refs:** [`../dead-code-registry.md`](../dead-code-registry.md) (the authoritative list), `../deviations-and-deferrals.md` DF1/DF5/DF7.
- **Why now (the answer to "when do we remove dead code"):** the removals **drop DB columns/models** (`Slot`, `SlotStatus`, `Booking.slotId`, `Trial.slotId`, possibly `Referral`). Dropping columns must go through **`prisma migrate`**, not the `db push` we've used (DF5). And it's only safe once nothing new will reintroduce a dependency — i.e., after all of P0–P5's features exist. So the cleanup is the **final pre-launch slice**, gated on the migration baseline.

## 3. Sequencing (from the registry §C — do not reorder)
1. **⚠️ FIRST rewrite the LIVE `include: slot` reads (review-caught — they're NOT all dead).** Several **live** read paths still `include: { slot: { select: { slotTime } } }` and would break when `Slot` is dropped — they must move to `trialAt` first: `trials/repo.ts:16,40,60` (student trial lists/detail), `studio/repo.ts:140` (academy trial detail), `payments/service.ts:239,259` (trial create + push — already `?? trialAt`, drop the `slot` side). Only `bookings/service.ts:103` is a true legacy fallback. Rewrite + test these as a **separate commit** so the Slot drop is purely mechanical.
2. **Generate the migration baseline (DF5) — NOT possible in this dev env.** `migrate dev` shadow-DB perms are **denied here** (DF5); a `prisma/migrations/` dir exists but every change used `db push`. So this step **must run in an environment with `migrate` perms** (pre-launch infra): generate the initial baseline + the Slot-drop migration there, commit the files, then the drop is auditable. **S5.4 cannot be completed in the current dev env** — it's a deliberate pre-launch, perms-gated task.
4. **A1 Slot system — one PR.** With the live reads rewritten (step 1), delete the module + routes + workers + api-client methods + seed/mock + schema model/enum/columns **together**. Migrate (step 2 env). Reseed.
5. **Orphaned `/trials` index** — confirm `trials/index.tsx` removed + the 3 repoints done (S5.1); **`trials/[id].tsx` stays**.
6. **A4 referral** — only after the product call.
7. **A5 legacy aliases** — last, after any screen/token migration.
8. **After each step:** `tsc` across all 5 packages **green**, full test suite **green**, and **round-trip the two flows that touched slots most — trial booking + the academy schedule** (they must still work off `trialAt`/`BatchTiming`).

## 4. Risks & guards
- **Blast radius:** A1 spans backend modules, workers, api-client, seed, schema, and a few repo reads — the registry lists every site. Removing one without the others won't compile; that's the safety net (the build tells you what's left).
- **Migrations vs `db push`:** every prior phase used `db push`; S5.4 is the moment to establish the migration baseline so production column-drops are auditable + reversible.
- **Don't over-delete:** `Academy.ratingCount` (DF7), `useSlots.ts` (rewritten adapter — live, not dead), and the dormant `Slot` *data* semantics that booking history might reference — verify no live read before dropping `slotId` columns.

## 5. Definition of Done

- [ ] The **Slot system is fully removed** (model + enum + columns + module + routes + workers + api-client + seed/mock) — `grep -ri slot src/` returns only intentional history, if any.
- [ ] The orphaned `/trials` screen is gone (S5.1); referral + legacy aliases handled per their decisions.
- [ ] **All 5 packages typecheck; full suite green; trial-booking + schedule round-trips pass.** The registry is updated (items struck through / file emptied).

After S5.4 the codebase is launch-clean: real migrations, no Slot system, no orphans. Update `dead-code-registry.md` to reflect what was removed and what (if anything) was deliberately kept (e.g., referral if the product chose dormant).
