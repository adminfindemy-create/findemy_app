## 3. The shared contract
```
respondToReview(...)  → after update, sendPushNotifications({ subjectId: review.userId, subjectRole:'student',
                          title:'Academy replied to your review', body:'…', data:{ screen:'academy', academy_id } })
createReview(...)     → after create, sendPushNotifications({ subjectId: academyId, subjectRole:'academy',
                          title:`${name} left a ${rating}★ review`, data:{ screen:'reviews' } })
worker class-reminder    → cron */15: project active batches' BatchTiming; in-studio sessions in the [25,35]-min
                            window AND not in SessionReminderSent(kind='class') → push enrolled students.
worker attendance-remind → same worker/table, kind='attendance', ~2h window → push students AND academy.
```
- **✅ Academy push resolution DONE (review-caught regression).** Pushes pass the **Academy id**; `sendPushNotifications` now resolves `academyId → AcademyAccount.id` internally (`push/repo.resolveAcademyAccountId`), and the bogus `User` FK on `ExpoPushToken.subjectId` — which blocked storing academy tokens at all — was dropped. New-review/review-reply academy pushes "just work" (callers pass `academyId`); this also fixed the pre-existing-broken "New trial booked" academy push.
- **⚠️ Durable idempotency — a `SessionReminderSent` table (review-caught; a Set is NOT enough).** Sessions are *projected* from `BatchTiming` (no row to mark) and an in-memory Set dies between worker runs → re-sends. Add `model SessionReminderSent { batchId, sessionDate @db.Date, kind, sentAt, @@unique([batchId, sessionDate, kind]) }`; the worker checks/creates this row per (batch, date, kind) — mirroring how `renewal-reminder` uses a durable counter and `CancelledSession` a unique row. Makes the worker safe against cron jitter / overlapping runs.
- **Cadence + window are coherent:** run every **15 min**; fire when a session start is in **[25,35] min** ahead (class) / **~2h** ahead (attendance) — wide enough to never miss between runs; the `SessionReminderSent` unique row prevents any double-fire.
- **Reuse the projection — export it.** `projectTrialTimes` (`batches/service.ts`) is **private**; export it (or a `projectBatchSessions` wrapper) so the worker + S3.4's schedule share one projection. Scan `BatchTiming`, not `Slot` (deleted in S5.4); filter `Batch.mode==='offline'` for attendance (online auto-marks).
- **Attendance reminder (owner decision: BOTH sides):** ~2h before an **in-studio** class session (from `BatchTiming`), push **enrolled students** ("Class today — check in with the QR") **and** the **academy account** ("Open the QR — your class is soon"). Online classes are excluded (no QR; they auto-mark on join). One worker computes the upcoming session once and fans out to both audiences with a per-(batch,date) sent-guard.
