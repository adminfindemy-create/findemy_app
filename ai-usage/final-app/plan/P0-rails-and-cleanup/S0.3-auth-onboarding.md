## 3. Shared contract
- Extend **`OnboardingRequest`** (`backend/shared/types/src/index.ts` — already exported) with:
  - `modes_offered: OfferedMode[]` — a **distinct** enum `OfferedMode = in_studio | online | home_visit` (labels "In-studio / Online / Home visit"). This is the academy's *offered* modes and is **separate from the batch delivery `Mode`** (S0.1) — `home_visit` is offered-only (no batch uses it), and the stored spelling of the in-studio/online values should follow the S0.1 mode-value decision so the two enums don't get conflated.
  - `tagline: string` (≤ ~80 chars).
- Endpoint: `POST /auth/onboarding` (or existing onboarding route) accepts the extended payload. `GET /studio/academy` returns `modesOffered` + `tagline`.

## 4. Component design
- **Backend:** add `modesOffered OfferedMode[]` to prisma `Academy` (**`tagline String?` already exists — reuse it, no new column**); migrate; persist both in the onboarding/academy service; return in academy reads.
- *(Note: the academy onboarding **UI** collects neither modes nor tagline today — `(auth)/onboarding.tsx` has only owner/academy/city/category/phone — so both inputs are new to that screen even though the `tagline` column exists.)*
- **Academy app:** `app/(auth)/onboarding.tsx` — add a **Modes offered** multi-select chip group (In-studio / Online / Home visit) + a **tagline** text field; include in the submit payload via `api-client`.
- **Student app:** REUSE welcome → phone → OTP → onboarding (learner profile) → interests. No change.
- **Shared:** the `OfferedMode` enum + extended `OnboardingRequest` live in `backend/shared/types`.
