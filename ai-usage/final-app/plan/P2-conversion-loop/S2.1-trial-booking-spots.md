## 1. Feature & intent
Book a paid trial against **`capacity − enrolled`** (no academy-published `Slot`s); reschedule **once**; refund if cancelled **≥4h** before; auto-confirm on payment. Trial **inherits the batch mode** and (via S2.2) carries a 4-digit attendance code. Retire the `Slot` model entirely.

- **Refs:** master-feature-document §3.4 (trial booking) / §2.4 (trial spots); prototypes booking slot→pay→confirm + academy Trials-today/trial-detail; **prototype decision (CLAUDE.md §7): trial spots are automatic = capacity − enrolled; the academy's "publish availability" screen was removed.**
- **Post-P1 reality (verified):** booking is **`POST /bookings {batch_id, slot_id}`** (`bookings/service.ts:6`) → reserve `Slot` → pay → **webhook creates the `Trial`** (`payments/service.ts:216`). Availability today = `slot.reservedCount >= slot.capacity` (`bookings/service.ts:19`). **4h-refund (`:105`) and reschedule-once (`:169`) are already correct** — they read `slot.slotTime`, to be repointed at `trial.trialAt`. `getBatchAvailability` (`batches/service.ts:53`) already returns `{capacity, enrolled, available, accepting}`.

> **⚠️ Where does the chosen time live? (verified gap)** Today the trial time exists **only** as `slot.slotTime`, reached via `Booking.slotId` — `Booking` has **no time column**. When slots go, the booking flow (`POST /bookings` → pay → webhook-creates-Trial) loses the time between steps. So **`Booking` must gain a `trialAt DateTime`** column: `POST /bookings` writes the student's chosen `trial_at` onto the Booking; the webhook reads `booking.trialAt` when creating the Trial (and for the 4h-refund/reschedule math). Without this column the webhook has no time to stamp on the Trial.

## 3. The shared contract (lock first)
```
GET /batches/:id/trial-availability
  → upcoming bookable trial datetimes derived from Batch TIMINGS (day_of_week + start_time),
    projected N weeks ahead, filtered to where (capacity − enrolled) > 0.
POST /bookings { batch_id, trial_at }   // NO slot_id; stores trial_at on Booking.trialAt (new col)
  → route schema CHANGES from { batch_id, slot_id } → { batch_id, trial_at } (coordinated app+backend break)
  → validate trial_at matches a Batch TIMING occurrence AND (capacity − enrolled) > 0  [headroom; trials don't consume — §4]
  → payment intent
  webhook(payment.captured) → create Trial { trialAt = booking.trialAt, attendanceOtp (per-user) }  // NO slotId
                            → booking 'confirmed'  → CommissionLedger (S2.4)
reschedule (booking-level rescheduleBooking, once) → new trial_at validated vs availability; off trial.trialAt
cancel (via booking) → refund if (booking.trialAt − now) ≥ 4h       // off booking.trialAt, not slot.slotTime
```
- **Trial mode = read-through, not denormalized.** Today `studio/service.ts:94` already emits the trial's mode by reading **`trial.batch.mode`** (the loaded batch relation) via the canonical map. So **do NOT add a `Trial.mode` column** — just ensure the `trials/service.ts` student serializer also includes `batch.mode` (canonical) the same way. *(Snapshot-at-booking is unnecessary for MVP — batches rarely change mode; flag only if that becomes a requirement.)*
- **Reschedule is booking-level (`rescheduleBooking`, `bookings/service.ts:159`), not `/trials/:id/reschedule`.** It currently takes a **`newSlotId`** and validates `newSlot.slotTime`/`newSlot.reservedCount` (`:173‑177`). **CHANGE it to take a `new_trial_at`**, validated against `GET /batches/:id/trial-availability` + headroom; write `booking.trialAt`; keep the reschedule-once guard (`:169`). This is a real signature change, not just a "repoint."
- **`trialAt` becomes canonical** (replaces `slot.slotTime` everywhere). Reschedule + 4h-refund repoint to it.
- **Rooms re-key (the gate):** online trial/class room id moves from `slot.id` → `batch.id` (one persistent Findemy-hosted room per batch — matches the prototype's per-batch live room; S3.3 refines). Do this **before** dropping `Slot`.

## 4. Availability semantics — RESOLVED (a trial does NOT consume a spot)
**Decision (locked with the product owner):** a booked trial **does not consume a permanent spot.** `trial_spots = capacity − active_enrollments` is **pure headroom** — "how many trials this batch can accommodate" — and only **enrolments** consume it. Therefore:
- **`getBatchAvailability` is UNCHANGED** (`capacity − enrolled`); the S1.3 `trial_spots` display is **unchanged**. Do **not** subtract trials.
- **Booking gate:** a trial may be booked whenever `capacity − enrolled > 0` (the batch has headroom). Booking a trial **does not decrement** `trial_spots`; multiple trials may share the same headroom (a trial is a one-time visit, not a seat).
- The master-plan E2E wording "spots decrement" is **superseded** — the academy's trial-spot number reflects **enrolments only**, so it stays stable when trials are booked and **drops when a trial converts to an enrolment** (S2.3).
