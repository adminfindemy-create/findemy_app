## 1. Feature & intent
Razorpay checkout already works for all four paid items (trial, enrolment, workshop, event) via a signed, **idempotent** webhook. This slice **adds a `CommissionLedger`**: one row per captured trial/enrolment recording Findemy's commission (taken from the **academy payout**, never added to the learner's price). It does **not** rebuild payments.

- **Refs:** master-feature-document §2.6 (payments); app-ideation `06-business-model` (commission now); gap-analysis (webhook idempotent = REUSE; commission ledger missing; platform-fee/credits already removed in S0.2).
- **Post-P1 reality (verified):** webhook routes 4 item types by `orderId`, each guarded by a `status==='captured'` early-return (idempotent). Trial created in the webhook (`payments/service.ts:216‑224`); enrolment activated (`payments/repo.ts:157‑167`), both inside `prisma.$transaction`. **No commission anywhere.** Platform-fee/credits fully gone (S0.2). `PaymentBreakdown = {base_paise, total_paise}`.

## 3. The shared contract (lock first)
```
CommissionLedger {
  id            cuid
  academyId     FK Academy
  itemType      enum: 'trial' | 'enrollment' | 'workshop' | 'event'
  itemId        string   // bookingId (trial) | enrollmentPeriodId | workshopPaymentId | eventPaymentId
  grossPaise    int      // the captured amount the commission is computed on
  commissionBps int      // rate snapshot at capture time (default COMMISSION_BPS)
  commissionPaise int    // floor(gross * bps / 10000)
  status        enum: 'accrued' | 'reversed'   // reversed on refund
  createdAt     DateTime
  @@unique([itemType, itemId])   // idempotency: re-delivered webhook can't double-write
}
```
- **Money rail:** all paise; `commissionPaise = floor(grossPaise * commissionBps / 10000)` (S0.1 bps convention, same rounding as `computeAmount`).
- **Commission is server-side only** — **not** in `PaymentBreakdown`, **not** added to the learner price (the learner pays `total_paise`; commission is deducted from the academy's payout). Confirm with the §"Open decision" on rate.
- **Idempotency (two layers):** (a) the webhook already early-returns if the payment is already `captured`, so the ledger write inside that branch runs at most once per real capture; (b) belt-and-braces `@@unique([itemType, itemId])` so a concurrent/re-ordered delivery can't double-insert. Write the ledger row **inside the same `$transaction`** as the trial-create / enrolment-activate so it's all-or-nothing.

## 4. Component design, per side
- **Backend**
  - `prisma/schema.prisma`: add `CommissionLedger` (above) + relation on `Academy`; `prisma db push`. Add a `CommissionItemType`/`CommissionStatus` enum (or string + zod) consistent with existing enum style.
  - **Central business config (NEW — see §"Config" below):** define `COMMISSION_BPS` in a single tunables module, **env-overridable** (`Number(process.env.COMMISSION_BPS ?? 1500)`), so the rate is changed in one place / via env without a code hunt. `computeCommission(grossPaise)` → `{ commissionBps, commissionPaise }` reads it. (Don't scatter the literal into the webhook.)
  - `payments/service.ts` capture branches: in the **trial** branch (`:213‑224`, inside the `$transaction`) and the **enrolment** branch (`:125‑128`, also inside a `$transaction`) write `CommissionLedger` via `repo.recordCommission(tx, …)` using `grossPaise = payment.amountPaise`. **`academyId` is already loaded in both branches** (verified): trial → `payment.booking.batch.academyId`; enrol → `epPayment.enrollmentPeriod.enrollment.batch.academy.id` (`payments/repo.ts:7` already includes `batch.academy`). **No query change needed.** `itemId = bookingId` (trial) / `enrollmentPeriodId` (enrol). (Workshop/event commission is **out of scope here** — the master contract says trials + enrolments; the enum carries the values, writing is a `TODO(S4.x)`.)
  - **Refund → reversal (⚠️ no refund webhook — verified):** `issueRazorpayRefund` (`payments/service.ts:19`) is **fire-and-forget** (calls Razorpay, returns a refund id; sets `refundStatus='requested'`). There is **no `refund.processed` webhook** — the webhook only handles `payment.captured`/`failed`. So reverse the commission **synchronously in each cancel path** that issues a refund, right after `issueRazorpayRefund` succeeds. **Three paths refund (all verified):** (1) **trial** cancel ≥4h (`bookings/service.ts`), (2) **workshop** cancel >24h (`workshops/service.ts`), (3) **enrolment** discontinue — `discontinueEnrollment` (`enrollments/service.ts:208‑226`) issues a **prorated or full** refund per the academy's `monthlyRefundPolicy`. **Reverse proportionally to the refund:** `reversedCommissionPaise = floor(commissionPaise * refundAmountPaise / grossPaise)` (full refund → fully reversed; prorated → partial), set `status='reversed'` (or a partial field). Idempotent (find-by-`itemId`; no-op if already reversed). *(If a refund webhook is added later for true settlement, move the reversal there.)*
  - `payments/repo.ts`: `recordCommission(tx, {...})` + `reverseCommission(itemType, itemId)`.
- **Academy app:** none in S2.4 — the academy's **net earnings** (gross − commission) is **S4.6** (`studio/repo.ts getEarnings:491` currently sums gross; S4.6 subtracts the ledger). Flag in the handoff.
- **Student app:** **REUSE** the 4 `pay.tsx` screens unchanged — amounts are already server-authoritative (fetched from the order endpoint; commission is invisible to the learner).
- **Shared:** add `CommissionLedger`/`CommissionItemType` types to `@findemy/types` if any read endpoint needs them (S4.6 will). `PaymentBreakdown` is **unchanged** (no commission field).

## Config — central tunables (NEW, addresses "is there a config file?")
> **AS-BUILT (deviation D2):** `@findemy/config` turned out to be a **tooling-config** package (ships `biome.json`/`tsconfig.base.json` only) — not a runtime source. So shared constants went into **`@findemy/types`** (the existing contract source) instead. See [`../deviations-and-deferrals.md`](../deviations-and-deferrals.md) D2.

There was **no business-config file**; `lib/env.ts` holds only **secrets/infra** env vars and tunables were **scattered inline literals**. The split, as implemented:
- **`@findemy/types` (shared — client + server must agree):** `DISCOUNT_BPS_CAP = 3000`, `OTP_LOGIN_LENGTH = 6`, `OTP_ATTENDANCE_LENGTH = 4`, `PLAN_MONTHS = {monthly:1, quarterly:3, annual:12}` (alongside the existing `PlanEnum`).
- **`backend/api/src/config.ts` (backend policy — never shipped to the client):** `COMMISSION_BPS` (env-overridable), `TRIAL_REFUND_MIN_HOURS = 4`, `WORKSHOP_REFUND_MIN_HOURS = 24`, `TRIAL_AVAILABILITY_WEEKS`.
> Split by audience so backend-only policy never leaks into the app bundle. Each P2 slice points its literal at the config: S2.4 → `COMMISSION_BPS`; S2.3 → `PLAN_MONTHS`/`DISCOUNT_BPS_CAP`; S2.2 → the OTP lengths; S2.1 → `TRIAL_REFUND_MIN_HOURS`/`TRIAL_AVAILABILITY_WEEKS`.
