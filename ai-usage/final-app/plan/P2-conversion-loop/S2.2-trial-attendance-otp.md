# S2.2 — Trial attendance OTP (4-digit) `[F12]` — CHANGE

> Feature Master Plan (master-execution-plan §3.2). Deps: **S2.1** (trial-detail reachable from Trials-today). Mechanical, but the **login and attendance OTPs share one generator** — splitting them is the whole risk.

## 1. Feature & intent
The student shows a **4-digit** attendance code; the academy enters/verifies it at the trial → present | no-show. Only the **attendance** OTP changes; the **login** OTP stays 6-digit.

- **Refs:** master-feature-document §2.2 (attendance) / §4.3 (academy attendance-otp); prototypes booking-detail OTP + academy attendance-otp; **prototype demo codes `4927` (Guitar) / `8260` (Vocals)** (CLAUDE.md §7).
- **Post-P1 reality (verified):** attendance OTP lives on **`User.attendanceOtp`** (per-user, **plaintext**, `schema VarChar(6)`), verified **timing-safe** at `trials/service.ts:119` (buffer-length-guarded, so a 4-vs-6 length mismatch fails safely). The verify route schema is `z.string().length(6)` (`trials/routes.ts:6`). **`generateOtpCode()` (`lib/otp.ts:4`) is shared** by the login OTP (`auth/service.ts:29`) **and** the attendance OTP (`auth/service.ts:83`, `:241`). `attendance.test.ts:52` asserts length 6.

## 3. The shared contract (lock first)
```
POST /studio/trials/:id/attendance { otp_code: string.length(4) } → present | no-show   (timing-safe)
User.attendanceOtp : 4-digit, per-user, plaintext (unchanged storage model; just shorter)
generateLoginOtp()      → 6-digit  (login/phone verification — UNCHANGED)
generateAttendanceOtp() → 4-digit  (set on user creation)
```
- **Two generators, not one** — the single shared `generateOtpCode()` is the bug waiting to happen: naively shrinking it to 4 digits would weaken **login** OTPs. Split first, repoint the two attendance call sites, leave login on 6.
- **Per-user, not per-trial** (decision, README §3): one 4-digit code per student, shown for all their trials. The prototype's `4927`/`8260` were per-trial demo values; with the per-user model they become **two seeded students' codes** (or the one demo student gets `4927`). Don't over-build a per-trial OTP for MVP.
- **Storage stays plaintext + timing-safe** (no hashing) — by design (permanent code, no expiry); the buffer-length guard already prevents 4-vs-6 cross-matches.

## 4. Component design, per side
- **Backend:** `lib/otp.ts` — keep `hashOtp`/`compareOtpHash`; replace `generateOtpCode` with `generateLoginOtp` (6) + `generateAttendanceOtp` (4) (or keep `generateOtpCode` as the login one + add `generateAttendanceOtp`). Source the lengths from the central config (`@findemy/types` `OTP_LOGIN_LENGTH = 6` / `OTP_ATTENDANCE_LENGTH = 4`) so the route schema length, the generators, and the app inputs all read one constant (see S2.4 §"Config"). Update `auth/service.ts:83,241`. Update `trials/routes.ts:6` schema length. `schema.prisma` `VarChar(6)→VarChar(4)` + `prisma db push`. Seed: demo student `attendanceOtp = '4927'` (4-digit, recognizable); generic users `String(1000 + seq).slice(0,4)`. `attendance.test.ts:52` → `toBe(4)`. `mock/fixtures.ts:18` → 4-digit (e.g. `'4927'`).
- **Academy app:** `attendance-otp.tsx` — `OTPInput length={6}→4`, the two `!== 6` checks → `!== 4`, and the "6-digit code" copy → "4-digit". Reachable from **Trials-today** (S2.1).
- **Student app:** `OTPDisplay.tsx` `Array.from({length:6})` → `{length:4}`; `trials/[id].tsx:255` hardcoded fallback `"742916"` → a 4-digit fallback (and prefer the real `attendanceOtp` from the auth store).
- **Shared:** none (no type change — it's a string).
